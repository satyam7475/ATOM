================================================================================
  2. EXECUTIVE SUMMARY
================================================================================

  ATOM is a Personal Cognitive AI Operating System: it perceives (voice,
  system context), understands (fast intent classification + optional LLM
  reasoning), decides (router with cache/memory/agentic path), acts (security-
  gated tools and desktop control), expresses (TTS + web UI), and learns over
  time (behavior, memory, autonomy — scope varies by module and config).

  Design center: offline-first local inference (GGUF LLM, faster-whisper,
  offline-capable TTS paths), single primary user (“Boss”), event-driven wiring
  via AsyncEventBus to keep modules replaceable (“every module is a replaceable
  organ”).

  Execution safety: tool calls and actions pass through SecurityPolicy and
  tiered confirmations where configured.

  Runtime options: (A) single Python process with asyncio and shared event bus;
  (B) V4 track — separate processes for STT/TTS/LLM/intent/context/decision/
  memory/brain orchestration, connected via ZMQ (see section 4.2).


================================================================================
  3. SYSTEM IDENTITY AND PRINCIPLES
================================================================================

  Principle                  | Implementation (summary)
  ---------------------------|--------------------------------------------------
  Offline-first              | Local GGUF LLM, faster-whisper STT, offline TTS
  Owner-centric              | Single owner; “Boss”; personal context models
  Event-driven               | AsyncEventBus; loose coupling between organs
  Security-gated             | SecurityPolicy on actions/tools
  Self-improving (scope)     | SelfEvolutionEngine, SelfOptimizer, trackers
  Modular organs             | Swappable subsystems behind stable contracts
  Cross-platform (intent)    | PlatformAdapter for OS differences
  System-aware               | SystemScanner, SystemWatcher, indexing
  Proactive (where enabled)  | JarvisCore, AutonomyEngine, predictions


================================================================================
  4. HIGH-LEVEL DESIGN (HLD)
================================================================================

  4.1 Logical architecture — rings (outer to inner)
  -------------------------------------------------

  Ring 1  PERCEPTION        STT, mic, system watcher, context, privacy filter
  Ring 2  UNDERSTANDING     IntentEngine (+12 sub-modules), CommandCache, Skills
  Ring 3  DECISION          Router, CacheEngine, MemoryEngine, ConversationMem,
                            ToolRegistry/Parser, ActionExecutor, Planner, Sandbox
  Ring 4  EXECUTION         *_actions, desktop_control, system/app/media/...
  Ring 5  EXPRESSION        TTS (SAPI/Edge), WebDashboard, personality/modes
  Ring 6  COGNITION         SecondBrain, goals, predictions, behavior, optimizer
  Ring 7  AUTONOMY          Autonomy, self-evolution, health, watchdog, behavior,
                            SecurityPolicy
  Ring 8  JARVIS LAYER      JarvisCore, OwnerUnderstanding, SystemScanner,
                            PlatformAdapter, SystemControl

  Backbone (all rings): AsyncEventBus

  ASCII overview:
  +-------------------------------------------------------------------------+
  | R1 Perception -> R2 Understanding -> R3 Decision -> R4 Execution        |
  | R5 Expression <- (responses, TTS, UI)                                   |
  | R6 Cognition / R7 Autonomy / R8 Jarvis intelligence (parallel layers)   |
  | AsyncEventBus = horizontal backbone                                       |
  +-------------------------------------------------------------------------+


  4.2 Runtime topology
  --------------------

  [Monolith — default integration story]
    main.py
      -> wires AsyncEventBus, StateManager, STT, Router, TTS, dashboard, ...
      -> all handlers share one asyncio loop (with thread pools where needed)

  [V4 multi-process — Cognitive OS track]
    run_v4.py orchestrates:
      core/ipc/zmq_broker.py          (broker)
      services/stt_worker.py
      services/tts_worker.py
      services/llm_worker.py
      services/intent_worker.py
      services/context_worker.py
      services/decision_worker.py
      services/memory_worker.py
      services/brain_orchestrator.py (Cognitive hub; log labels “V5” in script)
      main.py --v4                    (core)

    IPC helpers: core/ipc/zmq_bus.py, interrupt_manager.py, proxies.py

    The dedicated cognitive hub for the V4 track is
    services/brain_orchestrator.py — verify that entrypoint when changing V4.


  4.3 Main interaction pipeline (happy path)
  ------------------------------------------

  Microphone -> STT -> speech_final (event) -> Router
    -> (1) IntentEngine fast-path OR
    -> (2) CacheEngine / MemoryEngine OR
    -> (3) LLM agentic path: prompt + ToolRegistry -> stream -> ToolParser ->
        ActionExecutor (SecurityPolicy) -> ReAct loop as designed
    -> response events -> TTS / UI

  Parallel/background: SystemIndexer, MediaWatcher, HealthMonitor, autonomy
  hooks — via events and schedulers (see Blueprint for full catalog).


  4.4 Cross-cutting concerns
  --------------------------

  AsyncEventBus     Pub/sub; emit tiers (fast / normal / long) per Blueprint.
  StateManager      States: SLEEP, IDLE, LISTENING, THINKING, SPEAKING,
                    ERROR_RECOVERY (see Blueprint §6).
  SecurityPolicy    Central gate for tool/action authorization.
  Metrics / timing  MetricsCollector, PipelineTimer; priority scheduling for
                    voice vs background work.

================================================================================
  6. LOW-LEVEL DESIGN (LLD)
================================================================================

  6.1 AsyncEventBus (core/async_event_bus.py)
  -------------------------------------------
  - Subscribe: register async callbacks per event name.
  - Emit: queue events to a worker; isolate handler failures so one bad handler
    does not kill the bus.
  - Tiered emit methods per Blueprint (fast vs long-running handlers).

  6.2 Router — three layers
  -------------------------
  Layer 1: IntentEngine (~ms) — deterministic commands when patterns match.
  Layer 2: CacheEngine + MemoryEngine — exact/fuzzy and keyword memory.
  Layer 3: LLM agent — ToolRegistry-filled prompt, streaming, ReAct-style tool
  loop through ActionExecutor.

  6.3 Reasoning stack
  -------------------
  ToolRegistry     Static definitions and prompt-facing schemas (~40+ tools).
  ToolParser       Extract structured calls from model text (multiple formats).
  ActionExecutor   Validate tool + params, SecurityPolicy, confirmation flow,
                   dispatch to Python handlers; timeouts where applicable.
  ReasoningPlanner Template and LLM-generated multi-step plans.
  CodeSandbox      Restricted execution for math/helper eval.

  6.4 Representative algorithms / behaviors
  -----------------------------------------
  AudioPreprocessor (voice/audio_preprocessor.py): DC offset removal, pre-
  emphasis, spectral noise gate, peak normalization (see ATOM_LLD.md).

  CacheEngine: TTL-aware LRU plus Jaccard similarity for fuzzy hits.

  brain/memory_graph.py: MemoryNode (episodic/semantic/procedural), optional
  VectorMemory via ChromaDB when available; graph-style relationships.

  brain/behavior_model.py: UserState (focus, stress, mode, time), history
  lists for apps/commands/idle; informs proactive behavior.

  brain/reflection_engine.py: Periodic reflection over LearningEngine
  experiences; optional tie-in to planning_engine.

  MiniLLM / LocalBrainController: llama-cpp streaming, GPU offload config,
  sentence-buffer streaming to TTS (details in LLD + Blueprint §5).

  6.5 Persistence
  ---------------
  Primary persistence story in Blueprint: JSON under logs/ and config/ (no hard
  DB requirement for core operation). Vector memory / Chroma paths are optional
  (memory_graph). SecondBrain and MemoryEngine paths remain product-specific;
  align with settings on disk.

  [Voice command — simplified]

    Mic -> STTAsync -> speech_final -> Router._route
           |                           |
           |                           +-> IntentEngine -> action dispatch -> bus
           |                           +-> Cache/Memory hit -> response
           |                           +-> cursor_query -> LocalBrainController
           |                                 -> partial_response / tool loop
           |                                 -> ActionExecutor -> tools
           v
    StateManager / Metrics / Dashboard (parallel observers)

    SystemIndexer / MediaWatcher / SystemScanner -> context fusion / events
    HealthMonitor -> governor, recovery, snapshots
    AutonomyEngine -> suggestions (subject to policy thresholds)

================================================================================
  9. SECURITY MODEL (SUMMARY)
================================================================================

  - Air-gapped core philosophy: LLM/STT/TTS run locally; network tools are
    explicit capabilities subject to policy and user expectations.
  - Tiered tool risk: safe / moderate / dangerous / blocked (conceptual; exact
    names in SecurityPolicy and ToolRegistry).
  - Confirmations: dangerous operations may require explicit user confirmation
    through the personality/confirmation path.
  - Autonomy: high-confidence thresholds; destructive auto-execution avoided by
    engine rules — still passes SecurityPolicy when executed.

  10. EVOLUTION AND IMPROVEMENT AREAS (COLLABORATOR BACKLOG)
================================================================================

  From ATOM_V2_UPGRADE_PLAN.md “What Must Still Change” (still relevant as of
  this digest unless closed in a later revision):

  - Semantic memory: move beyond pure keyword overlap JSON to meaning-based
    retrieval at scale (embedding/vector strategy is partially explored in
    brain/memory_graph.py — productize and unify).
  - Document knowledge: ingestion pipeline for PDFs/docs/notes (Blueprint gap).
  - Wake word: reduce always-on STT cost; OpenWakeWord-class option in plan.
  - Owner verification: stronger than config-only identification.
  - HLD note: reasoning and tool use called out as addressed in v18; keep tests
    and docs in sync when bumping versions.

  From ATOM V4 Cognitive OS blueprint (directional):

  - “LLM last, not first” — push more through intent, cache, and memory graph.
  - Memory graph + episodic/semantic/procedural structure as first-class.
  - Proactive engine + skill macros for repeatable multi-tool workflows.

  Process architecture: V4 multi-process improves isolation and may complicate
  debugging — invest in observability (logging correlation across workers).
